# 工艺路线模块 (Process Route Module) 设计文档

## 1. 背景与目标

### 问题背景
Wms.Core 是一个电池生产 WMS 系统，托盘（Unitload）在多个工序间流转（化成、分容、OCV3/4 等）。当前工序推进逻辑**硬编码**在 `UnitloadService.cs` 第 855-872 行的 `GetNextOperation()` 方法中，通过枚举值+1 和两个特殊映射实现。这种方式无法支持不同产品走不同路线、分支选择、工艺调整等业务需求。

### 设计目标
1. **按物料匹配路线** — 不同物料绑定不同工艺路线，托盘创建时自动匹配
2. **有向图路线结构** — 支持一对多/多对一的分支与汇合
3. **人工分支选择** — 分支节点由操作员在前端选择下一步走向
4. **版本管理（快照锁定）** — 路线修改创建新版本，运行中托盘走旧版本
5. **双模式兼容** — 保留硬编码作为后备，存量托盘不受影响
6. **可视化** — 前端用 G6 展示路线图，托盘位置高亮

### 非目标
- 不关联具体库位/设备（库位分配仍由现有 `LocationAllocationEngine` 负责）
- 不做基于电芯测试数据的自动条件评估（分支仅人工选择）
- 不重构现有流程引擎（仅适配/新增 3 个节点：`AdvanceOperationHandler` / `ProcessTagVerificationHandler` / `VerifyProcessStepsHandler`）

## 2. 数据模型

### 2.1 实体关系结构

```
ProcessRoute (1) ──< (N) ProcessRouteVersion ──< (N) ProcessRouteStep
                            │                        │
                            └──< (N) ProcessRouteTransition (FromStepId/ToStepId → Step)
ProcessRoute (1) ──< (N) ProcessRouteMaterialBinding >── (1) Materials
Unitload (1) ──< (N) UnitloadProcessRouteLog
```

### 2.2 新增实体

| 实体 | 表名 | 核心字段 |
|------|------|---------|
| `ProcessRoute` | `ProcessRoutes` | ProcessRouteId, Code(唯一), Name, CurrentVersion, IsActive, IsBuiltIn, Priority |
| `ProcessRouteVersion` | `ProcessRouteVersions` | Id, ProcessRouteId, Version, Status(Draft/Published/Archived), ChangeLog, PublishedTime |
| `ProcessRouteStep` | `ProcessRouteSteps` | Id, VersionId, OperationCode(对应工序枚举名), DisplayName, StepType(Start/Normal/End), IsStart, IsEnd, SortOrder |
| `ProcessRouteTransition` | `ProcessRouteTransitions` | Id, VersionId, FromStepId, ToStepId, TransitionType(Sequential/Branch), Label, IsDefault, SortOrder |
| `ProcessRouteMaterialBinding` | `ProcessRouteMaterialBindings` | Id, ProcessRouteId, MaterialId, Priority, IsActive |
| `UnitloadProcessRouteLog` | `UnitloadProcessRouteLogs` | Id, UnitloadId, VersionId, StepId, OperationCode, ActionType(Enter/Advance/BranchSelect), FromOperation, ToOperation, SelectedTransitionId, Operator |

### 2.3 修改现有实体

`Unitload.cs` 新增 5 个字段：
- `ProcessRouteId` (int?) — 绑定的路线（null=硬编码模式）
- `ProcessRouteVersionId` (int?) — 锁定的版本（快照引用）
- `CurrentStepId` (int?) — 当前步骤
- `NextStepId` (int?) — 下一步骤（分支等待时为 null）
- `IsAwaitingBranchSelection` (bool?) — 是否等待人工选择

### 2.4 值对象

`ProcessRouteGraph.cs` — 路线图内存结构，提供 `GetStartStep()`、`GetOutgoingTransitions(stepId)`、`FindStepByOperation(code)`、`IsBranchNode(stepId)` 等查询方法。

## 3. 物料与路线的关联设计

### 3.1 多对多绑定表

通过 `ProcessRouteMaterialBinding` 关联表实现物料与路线的多对多关系：
- **一个物料可绑定多条路线**（不同 Priority），匹配时取优先级最高的
- **一条路线可绑定多个物料**（如同一产品线的不同批次物料共用一条路线）
- `Priority` 字段（int，数值越大优先级越高）解决多路线冲突
- `IsActive` 字段控制绑定是否生效

### 3.2 匹配逻辑

```csharp
// 1. 查询物料的绑定记录（按 Priority 降序）
var binding = await _db.ProcessRouteMaterialBindings
    .Where(b => b.MaterialId == materialId && b.IsActive)
    .Join(_db.ProcessRoutes.Where(r => r.IsActive),
        b => b.ProcessRouteId, r => r.ProcessRouteId,
        (b, r) => new { Binding = b, Route = r })
    .OrderByDescending(x => x.Binding.Priority)
    .FirstOrDefaultAsync();

// 2. 获取该路线的最新 Published 版本
var version = await _db.ProcessRouteVersions
    .Where(v => v.ProcessRouteId == binding.Route.ProcessRouteId
        && v.Status == "Published")
    .OrderByDescending(v => v.Version)
    .FirstOrDefaultAsync();

// 3. 缓存结果（materialId → versionId，10分钟滑动过期）
```

### 3.3 多物料托盘处理

取**第一个 UnitloadItem 的 MaterialId** 作为主物料进行路线匹配。

### 3.4 绑定流程 (BindRouteAsync)

1. 根据 MaterialId 匹配路线版本
2. 设置 ProcessRouteId / ProcessRouteVersionId
3. 根据 CurrentOperation 定位当前步骤
4. 计算下一步（单出边直接设置，分支节点取默认分支或等待人工选择）

## 4. 版本管理策略（快照锁定）

**核心原则**：运行中托盘走旧版本，新托盘走新版本。

| 操作 | 行为 |
|------|------|
| 修改路线步骤/分支 | 基于当前 Published 版本创建新的 Draft 版本，不影响在线托盘 |
| 发布新版本 | 新版本变为 Published，原 Published 版本变为 Archived |
| **运行中托盘** | `ProcessRouteVersionId` 已锁定，继续走绑定的旧版本 |
| **新创建托盘** | `MatchRouteByMaterialAsync` 返回最新 Published 版本 |
| 归档旧版本 | 状态变 Archived，但已绑定该版本的托盘仍可继续走 |

**版本状态流转**：Draft → Published → Archived

**关键实现**：
- `GetRouteGraphAsync` 查询路线图时**不限制 Version Status**
- `MatchRouteByMaterialAsync` 只返回 `Status == "Published"` 的版本

## 5. 双模式兼容设计

### 5.1 依赖关系

```
UnitloadService ──→ IProcessRouteService（创建托盘时调用 BindRouteAsync）
AdvanceOperationHandler ──→ IUnitloadService + IProcessRouteService（双模式判断）
ProcessTagVerificationHandler ──→ 无外部依赖（读 Unitload.CurrentOperation 比较多托盘工序一致性）
VerifyProcessStepsHandler ──→ 无外部依赖（读 Unitload.OperationNumber 比较多托盘工艺次数）
ProcessRouteService ──→ 无依赖其他业务服务（独立）
```

### 5.2 推进逻辑

```
if (unitload.ProcessRouteVersionId.HasValue)
    → 走路线引擎 (AdvanceOperationAsync)
    → 若失败回退硬编码
else
    → 走硬编码 (GetNextOperation)
```

### 5.3 缓存策略

参照 FlowEngineService 模式：
- `static int _cacheVersion` + `IMemoryCache`
- 缓存键：`processroute:graph:v{version}:{versionId}`
- 滑动过期 10 分钟
- `ClearCache()` 使用 `Interlocked.Increment` 使旧键失效

## 7.5 分层归属（与底层封装方案对齐）

> 与 [WMS底层封装与平台-项目分离方案.md](WMS底层封装与平台-项目分离方案.md) 第二章"关键决策：工艺路线模块"配套。

| 内容 | 所在层 |
|------|--------|
| 实体（6 个 + ProcessRouteGraph 值对象） | 底层 platform（随 Domain） |
| IProcessRouteService 接口 | 底层 platform（Phase 2 时随 Application.Contracts） |
| ProcessRouteService 实现 | 项目层 project（Infrastructure） |
| 6 个 EF Configuration | 项目层 project（Infrastructure） |
| ProcessRouteController | 项目层 project（WebApi） |
| ProcessRouteSeeder + DbInitializer 建表 SQL | 项目层 project（WebApi） |
| 3 个 NodeHandler（AdvanceOperation / ProcessTagVerification / VerifyProcessSteps） | 底层 platform（Phase 3 时随 Engine） |

**客户特异性通过数据配置实现**：电池行业工序（化成、分容、OCV3/4 等）只在 Seeder 种子数据中体现，新客户通过覆盖项目层 Seeder + 修改 MaterialBinding 表即可定制，无需触碰底层 .cs。
